    FKServer2 Asset Pipeline Mapping#

    Although the complete client source code for FKServer2 is not available, we can accurately reverse-engineer how the extracted visual assets (like 3D meshes, map grids, and UI icons) were originally connected to the game logic by analyzing the server source code (\FKServer2\FKSvr2Server).

    The architecture of FKServer2 (which is based on the TLBB engine) is heavily data-driven. Both the server and the client share a master set of .txt\ and .ini\ configuration tables. The server uses these tables to load statistical logic, while the client uses them to map identifiers to visual asset filenames.

    The master list defining the paths for these tables is found in the server code at \FKSvr2Server\Base\FileDef.h.

    1. Map and Environment Assets#

    Map geometries and level boundaries are mapped using the \SceneInfo.ini\ configuration file.

    • Config File: \SceneInfo.ini\ (Mapped by \FILE_SCENE_INFO\ in \FileDef.h)
    • C++ Struct: _SCENE_DATA\ (Parsed in \FKSvr2Server\Base\Config.cpp)
    • Mapping Mechanism: The server iterates through sections labeled [scene0], [scene1], etc. It binds an internal integer \SceneID\ to specific map data via the \ ile=\ key. For example, \ ile=dali.scene\ tells both the server and client which base mesh/nav data to load for that zone. If you have extracted .scene\ or .nav\ assets, they align perfectly with the \ ile\ entries in \SceneInfo.ini.

    2. NPC, Monster, and Player Models#

    Characters and monsters are mapped primarily using the \MonsterAttrExTable.txt\ and hair geometry tables.

    • Config Files: \MonsterAttrExTable.txt\ (\FILE_MONSTER_ATTREX), \CharHairGeo.txt\ (\FILE_HAIRSTYLE)
    • C++ Structs: \MONSTER_EXT_ATTR\ (Found in \GameTable.h)
    • Mapping Mechanism: \MonsterAttrExTable.txt\ contains the definitions for NPCs and enemies. Crucially, the \MONSTER_EXT_ATTR\ struct defines a field called \m_OutLook. The \m_OutLook\ integer is the visual identifier. The client uses this \OutLook\ ID to query a separate client-side table (often \CharModel.txt\ or similar) to fetch the actual .mesh\ or .skeleton\ file for rendering. Similarly, player hairstyles map directly via \CharHairGeo.txt.

    3. Items, Equipment, and UI Icons#

    Inventory items, weapons, and spell icons rely on base value tables.

    • Config Files: \CommonItem.txt\ (\FILE_COMMON_ITEM), \WhiteEquipBase.txt\ (\File_WHITE_EQUIP_BASE), etc.
    • Mapping Mechanism: The server identifies an item by an integer ID. When an item is generated, its properties are referenced from these .txt\ files. The client loads the exact same text files to lookup the item’s \IconID\ (for the inventory UI) or its 3D \MeshID\ (when equipped on a character or dropped on the ground).

    Tip: If you are looking to reuse the extracted .mesh, .skeleton, or .dds\ UI icon files in the new \�_ath_client, you should parse the .txt\ configuration files from the server’s \Public/Config\ directory to build an automated lookup dictionary in Go.
